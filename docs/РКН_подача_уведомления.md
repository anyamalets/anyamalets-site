## 7. Проверка кода сайта (на 2026-05-01)

Проверены: `app/**/*.tsx`, `sanity/schemas/*.ts`, `sanity/client.ts`,
поиск форм, сабмитов, fetch-вызовов и embed-виджетов.

**Sanity:**
- Единственная схема — `post` (Статья: title, slug, body, coverImage).
  Полей под имя/телефон/e-mail клиента нет.
- `sanity/client.ts` — клиент только на чтение из dataset `production`,
  без ввода клиентских данных.
- Studio (`/studio`) — авторизуется только сама оператор, ПД клиентов
  туда не попадают.

**Вывод:** ПД клиентов в Sanity не передаются. Sanity используется
только для контента блога. Трансграничной передачи ПД через CMS нет.

⚠️ Условие сохранения этого вывода: в `body` статей не должно быть
ФИО клиентов, цитат с идентифицирующими подробностями, фото клиентов.
Это и так запрещено клинической этикой (см. вашу установку: запрет
на клинические кейсы в контенте), но если в схему когда-то добавится
поле «отзывы клиентов» с ФИО — статус Sanity по 152-ФЗ изменится.

**Формы на сайте:**
- `<form>`, `onSubmit`, `fetch()` к собственному backend — **отсутствуют**.
  Своих API-роутов для приёма заявок нет.
- Запись клиентов: iframe-виджет Planerka (`PlanerkaEmbed.tsx`) —
  ПД (имя, телефон, e-mail) уходят напрямую в planerka.app в обход
  вашего сайта. Серверы Planerka в РФ — это уже учтено в политике.
- Альтернативные точки контакта: Telegram (@anya_malets, https://t.me/...)
  и `mailto:psy@anyamalets.ru`. Telegram — отдельная история (см. ниже).

**Telegram (t.me/anya_malets):**
- Сайт даёт **только ссылки** на Telegram, не embed-виджет. Технически
  ПД через сайт не передаются.
- В переписке через Telegram, согласно `/consent` и
  `/agreement-health-data`, **не указываются** ФИО, диагнозы, сведения
  о состоянии. Telegram используется только для оргкоммуникации
  (напоминания о времени, ссылки на материалы). Сведения о здоровье
  явным условием в мессенджеры не передаются.
- Telegram не выступает обработчиком по поручению: нет CRM-интеграции,
  нет бота-приёмника заявок, базы клиентов в Telegram не
  систематизируются. Это личный мессенджер как коммуникационный
  канал — аналогично звонку через сотового оператора.
- **Вывод:** трансграничной передачи ПД в смысле ст. 12 ФЗ № 152-ФЗ
  не происходит. Отдельное уведомление РКН по Telegram не требуется.
  В основной форме уведомления Telegram не указывать как получателя.
- **Рекомендация по укреплению позиции:** добавить в политику
  `/privacy-policy` раздел 7 явный пункт о статусе Telegram —
  «коммуникационный канал, без накопления и систематизации ПД,
  без передачи сведений о здоровье и идентифицирующих данных».
  Это защитит при проверке РКН.

**Яндекс.Метрика:**
- Серверы в РФ. ID счётчика стоит проверить и указать в уведомлении
  по запросу РКН (в основной форме обычно не требуется).

**Хостинг сайта (Vercel):**
- Vercel получает технические данные посетителей (IP, User-Agent,
  заголовки) как любой CDN — это побочный эффект маршрутизации
  трафика, не «сбор ПД» в смысле формирования базы клиентов.
- Локализация по ч. 5 ст. 18 ФЗ № 152-ФЗ применяется к базам данных,
  в которых ПД систематизируются по субъекту (CRM, клиентская база).
  Логи CDN — не такая база; у Vercel нет привязки IP к ФИО клиента.
- Через Vercel **не проходят** формы с именем/e-mail/телефоном —
  своих API-роутов для приёма заявок нет, лиды уходят прямо в Planerka
  iframe в обход Vercel.
- **Вывод:** уведомление о трансграничной передаче из-за Vercel
  не требуется. Утверждение в политике «передача за пределы РФ
  не осуществляется» остаётся корректным.
- Опционально: упомянуть Vercel в разделе 7 политики как технического
  CDN-провайдера для прозрачности — это не меняет статус по 152-ФЗ,
  но снимает вопросы, если их когда-то зададут.
